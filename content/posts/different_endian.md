我们都知道，CPU 中寄存器的数量是极其有限的。当一个结构的元素数量已经远超出寄存器所能容纳的大小时，超出寄存器的部分数据需要被存放在储存器中，等待控制器下达数据传送指令，才能把后续的数据存放到寄存器中。结构和数组一般存放在存储器中。

寄存器是固定的，例如 MIPS 架构限制寄存器总共有 32 个。在汇编代码中可以通过不同的标识符来访问，例如访问 `$s0` 寄存器。

存储器中存放了大量的数据，该如何从存储器中取出数据存放在寄存器中成为了问题。如何知道所需的数据在存储器的什么地方？

数据在储存器中会有各自的地址，可以将储存器理解为一个巨大的一维数组（一大片连续空间）。就像酒店房间一样，酒店房间的序号是恒定不变的，但是其中的数据可以变化。这个为数据选定房间的操作，就叫编址。

当多字节数据存储在内存中时，还会涉及到字节的排列顺序问题，这称之为端序，常见的端序有大端序和小端序。大端序将最高有效字节存放在低地址处（最左侧），小端序将最低有效字节存放在最低处。

例如有一个 32 位整数 `0x12345678`，这就需要使用 4 个字节来存储。当使用小端序，数据在内存中的排列顺序如下：

```goat
Little end               Big end
------------------       ------------------
| Address | Data |       | Address | Data |
| 1000    | 78   |       | 1000    | 12   |
| 1001    | 56   |       | 1001    | 34   |
| 1002    | 34   |       | 1002    | 56   |
| 1003    | 12   |       | 1003    | 78   |
------------------       ------------------
```

小端序更加符合 CPU 运算的处理习惯，更加符合硬件直觉，所以硬件实现更加容易。

网络协议使用大端序，大端序更加适合人类阅读。

以 MIPS 为例，MIPS 按字节编址，所以字的起始地址必定是 4 的倍数。这也称之为对齐限制，在开发中，也称为内存对齐，通过内存对齐能够加快数据传输。CPU 每读取一次的数据宽度是 4 字节，如果不对齐，CPU 需要额外再读一次并拼接，产生了额外的开销。

现代 CPU 如 MIPS 和 x86-64 基本上都是按字节编址，即每个地址就是 1 字节。例如读取一个 32 位的整数，CPU 就会读四个地址，为什么是四个？

```goat
32 bit = 4 byte
```
